# QuickJob Applier — Caching Architecture

This document explains every caching layer in the system, where it lives, when it fires, and how much it saves. The goal is to consume LLM tokens **only when no cheaper alternative exists**.

---

## Why Caching Matters

The default Groq free tier gives **100,000 tokens per day**. Without caching, a single full session (upload → search → 5 previews → apply) consumes roughly 40,000–60,000 tokens. With caching warm, the same session uses under 5,000 tokens — mostly because preview calls hit the cache.

---

## The 3-Layer Stack

Every operation that could call an LLM passes through three layers in order:

```
┌─────────────────────────────────────────────────────────────┐
│  Layer 1 — Memory files  (0 tokens, 0 API calls)            │
│  Flat JSON files loaded once at startup.                     │
│  Handles: job title taxonomy, ATS rules, cover templates,   │
│  system prompts, spell corrections, synonyms.               │
└─────────────────────────────────────────────────────────────┘
         │ miss (skills not in taxonomy, score ambiguous)
         ▼
┌─────────────────────────────────────────────────────────────┐
│  Layer 2 — ChromaDB semantic cache  (0 tokens on hit)       │
│  Persists across restarts. Three separate collections.      │
│  Handles: LLM responses, resume previews, job titles.       │
│  Hit threshold: 88–92% cosine similarity.                   │
└─────────────────────────────────────────────────────────────┘
         │ miss (new content, below similarity threshold)
         ▼
┌─────────────────────────────────────────────────────────────┐
│  Layer 3 — LLM with context compaction  (tokens consumed)   │
│  Response stored in Layer 2 immediately for future reuse.   │
│  Prompt truncated to minimum required size before sending.  │
└─────────────────────────────────────────────────────────────┘
```

## Layer 1 — Memory Files

Location: `quick_job_backend/memory/`

These files are loaded once when the server starts and stay in memory. No API calls, no tokens.

### `system_prompts.json`

Stores the system prompt for every LLM call type. Edit this file to tune behaviour globally without touching code.

```json
{
  "experience_rewriter": "You are a professional resume writer...",
  "ats_scorer":          "You are an ATS (Applicant Tracking System) expert...",
  "cover_letter_writer": "You are an expert cover letter writer...",
  "study_plan_generator": "You are a career coach...",
  ...
}
```

**Token saving:** ~50 tokens per call — system prompts are not regenerated by the LLM, they are injected from disk.

---

### `job_title_taxonomy.json`

Maps skill signals → job titles by seniority bracket. When a candidate's skills match 2+ signals in a category, titles are resolved entirely without LLM.

```
Skills: ["pytorch", "transformers", "langchain", "mlops"]
  → category: data_science_ml (4 signal matches)
  → bracket: 6-9 years
  → titles: ["Senior Data Scientist", "Lead Data Scientist", "Senior ML Engineer"]
```

Seniority brackets: `0-2` / `3-5` / `6-9` / `10+` years

**Token saving:** Entire `job_title_extractor` LLM call avoided — roughly 800–1,200 tokens per session.

---

### `job_title_synonyms.json`

Two purposes:

1. **Synonym groups** — when the candidate's current title is found in a group, all other titles in that group are also suggested:
   ```
   "Data Scientist" → also suggests:
   "Applied Scientist", "ML Scientist", "Data Science Engineer"
   ```

2. **Misspelling corrections** — applied on the UI before any API call:
   ```
   "Senoir Engieer" → "Senior Engineer"  (corrected client-side, 0 tokens)
   ```

---

### `ats_rules.json`

Rule-based ATS scoring for clear-cut cases. The LLM is only called for scores in the ambiguous 30–80% range.

```
skill match > 80%  →  score = 75 + (match_pct / 5)   [rule-based, 0 tokens]
skill match < 30%  →  score = match_pct               [rule-based, 0 tokens]
skill match 30-80% →  LLM called                      [tokens consumed]
```

**Token saving:** ~600 tokens per ATS call avoided when match percentage is clear.

---

### `cover_letter_templates.json`

Pre-defined structure (opening → body → close) and forbidden opener phrases loaded before the LLM call. The LLM fills the content, not the structure.

**Token saving:** ~100 tokens per cover letter (structure not re-generated).

## Layer 2 — ChromaDB Semantic Cache

Location: `quick_job_backend/vector_db/`

Three separate ChromaDB collections, each with its own similarity threshold and key strategy.

---

### Collection 1 — LLM Response Cache

**Path:** `vector_db/llm_cache/`  
**Module:** `llm_cache.py`  
**Similarity threshold:** 88%  

Every single LLM call in the system goes through `cached_llm_call()`. This is the universal cache.

#### How it works

```
1. Build cache key = SHA-256 of (system_prompt + user_prompt)
2. Exact hash lookup  →  hit: return instantly (0 tokens)
3. Embed the prompt   →  semantic search in ChromaDB
4.   similarity ≥ 88% →  return cached response (0 tokens)
5.   similarity < 88% →  call LLM, store result, return
```

#### What is cached

| Prompt type | Typical size | Max (after compaction) |
|-------------|-------------|----------------------|
| `experience_rewrite` | 3,000–5,000 chars | 4,000 chars |
| `skills_rewrite` | 1,500–3,000 chars | 2,000 chars |
| `ats_score` | 2,000–4,000 chars | 2,500 chars |
| `cover_letter` | 2,500–4,000 chars | 2,000 chars |
| `study_plan` | 2,000–3,500 chars | 2,000 chars |
| `jd_match` | 2,000–4,000 chars | 3,000 chars |
| `resume_parse` | 5,000–8,000 chars | 6,000 chars |
| `job_title_extractor` | 3,000–6,000 chars | 1,500 chars |

#### Example

```
First preview: "Senior Data Scientist @ OCBC Singapore"
  → experience_rewrite: LLM called, response cached

Second preview: "Lead Data Scientist @ OCBC Singapore"  (same JD, slightly different title)
  → experience_rewrite: prompt is 91% similar → cache hit, 0 tokens
```

---

### Collection 2 — Resume Preview Cache

**Path:** `vector_db/` (default ChromaDB path)  
**Module:** `resume_cache.py`  
**Similarity threshold:** 92%  

Caches the complete output of the preview pipeline per job: resume text, cover letter, ATS score, study plan, and edit summary.

#### Cache key strategy

```
Exact key  = SHA-256(resume_json[:2000] + job_title + company + jd[:500])
Embed text = "Job: {title} at {company}. JD: {jd[:300]}. Skills: {skills}. Exp: {N}yrs"
```

#### What is stored per entry

```json
{
  "resume_path":     "job_apply_output/20260321_143022_OCBC/resume.pdf",
  "cover_path":      "job_apply_output/20260321_143022_OCBC/CoverLetter_OCBC.pdf",
  "resume_txt_path": ".../.txt",
  "cover_txt_path":  ".../.txt",
  "ats_score":       "94",
  "match_score":     "87",
  "verdict":         "Strong match",
  "matched_skills":  "[\"Python\", \"ML\", \"SQL\"]",
  "missing_skills":  "[\"Spark\"]",
  "study_plan":      "{...full JSON...}",
  "edit_summary":    "{\"changes\": [{\"original\": \"...\", \"updated\": \"...\", \"reason\": \"...\"}]}"
}
```

#### Example

```
Click "Preview Tailored Resume" on OCBC Data Scientist job:
  First click:  cache miss → LLM pipeline runs → ~8,000 tokens consumed → result cached
  Second click: exact hash hit → response in <100ms, 0 tokens
  
Next week, same resume + similar OCBC job posting:
  92% similarity hit → same cached documents reused
```

### Collection 3 — Job Title Cache

**Path:** `vector_db/title_cache/`  
**Module:** `memory_manager.py`  
**Similarity threshold:** 92%  

Caches job title suggestions keyed by resume fingerprint.

#### Resume fingerprinting

```python
fingerprint = SHA-256(
    name.lower() +
    email.lower() +
    sorted(skills[:20]).join(",") +
    sorted(companies[:5]).join(",")
)
```

This means: same person uploads their resume next week with minor formatting changes → **same fingerprint** → instant title cache hit.

#### Example

```
Upload "pinnitigurunaidu-resume-v5.pdf"
  → fingerprint: a3f9c2d1b4...
  → taxonomy resolves: ["Senior Data Scientist", "Lead Data Scientist", ...]
  → stored in title cache

Upload "pinnitigurunaidu-resume-v6.pdf" (same skills, updated dates)
  → fingerprint: a3f9c2d1b4...  (identical — dates not in fingerprint)
  → title cache exact hit → 0 tokens
```

## Context Compaction

Before any LLM call, `llm_cache._compact_messages()` truncates prompt content to the minimum required for that task. This reduces token usage even when there is a cache miss.

```
Full resume JSON:     ~4,000–8,000 tokens
After compaction:
  "titles" focus:     ~200  tokens  (name + current role + skills only)
  "experience" focus: ~600  tokens  (last 3 jobs only)
  "skills" focus:     ~200  tokens  (skills + certifications only)
  "match" focus:      ~400  tokens  (skills + 2 recent titles + exp years)
```

JD text is also compacted: `compact_jd()` strips equal-opportunity boilerplate, benefits sections, and company background before sending to LLM. A 3,000-character JD typically compresses to under 800 characters.

## Full Session — Token Flow

```
Session: upload resume → search 2 portals → preview 5 jobs → apply 3

STEP 1 — Upload + Title Extraction
  resume_parse:         ~2,000 tokens  (LLM call, then cached)
  job_title_extractor:  0 tokens       (taxonomy hit for data science resume)
  job_identifier:       0 tokens       (portal list from COUNTRY_PORTALS dict)

STEP 2 — Job Search
  search:               0 tokens       (HTTP API calls to Adzuna + Serper)
  jd_match × 20 jobs:   0 tokens       (keyword scoring, no LLM)

STEP 3 — Preview × 5 (first session, cold cache)
  jd_match × 5:         ~1,000 tokens  (cached after first similar call)
  experience_rewrite × 5: ~3,500 tokens (compacted, cached per job)
  skills_rewrite × 5:   ~1,000 tokens  (compacted, cached per job)
  ats_score × 5:        0–500 tokens   (rule-based when clear, LLM otherwise)
  cover_letter × 5:     ~2,500 tokens  (template structure pre-loaded)
  study_plan × 5:       ~2,000 tokens  (compacted)

STEP 4 — Apply × 3
  resume + cover reuse: 0 tokens       (previewed files reused directly)
  
TOTAL (cold cache):     ~12,500 tokens
TOTAL (warm cache):     ~2,000 tokens  (only parse + any new jobs)
```

## Embeddings Strategy

The embedding function used for semantic search is resolved at startup in this order:

| Provider | Model | Cost | Quality |
|----------|-------|------|---------|
| OpenAI (if `OPENAI_API_KEY` set) | `text-embedding-3-small` | ~$0.00002/query | Best |
| Gemini (if `GEMINI_API_KEY` set) | `models/embedding-001` | Free tier | Good |
| Hash fallback (always available) | 256-dim hash vector | Free | Exact match only |

The hash fallback still provides exact-match caching (SHA-256 key lookup). Semantic similarity search (finding 92% similar resumes) only works with OpenAI or Gemini embeddings.

**Recommendation:** Add a free `GEMINI_API_KEY` to get semantic similarity at no cost.

## Cache File Locations

```
quick_job_backend/
├── memory/                          # Layer 1 — flat JSON, loaded at startup
│   ├── system_prompts.json          # Reusable LLM system prompts
│   ├── job_title_taxonomy.json      # Skill → title mapping (8 categories)
│   ├── job_title_synonyms.json      # Title synonyms + misspelling corrections
│   ├── ats_rules.json               # Rule-based ATS scoring thresholds
│   └── cover_letter_templates.json  # Cover letter structure templates
│
└── vector_db/                       # Layer 2 — ChromaDB, persists across restarts
    ├── llm_cache/                   # All LLM response cache (88% similarity)
    ├── title_cache/                 # Job title cache per resume fingerprint
    └── (root)                       # Resume preview cache (92% similarity)
```

All ChromaDB data persists across server restarts. Memory files are plain JSON — edit them directly to tune behaviour without restarting (they reload on the next request).

## Clearing the Cache

To clear a specific cache:

```bash
# Clear LLM response cache
rm -rf ~/quick_job_backend/vector_db/llm_cache

# Clear resume preview cache  
rm -rf ~/quick_job_backend/vector_db/chroma.sqlite3
rm -rf ~/quick_job_backend/vector_db/[uuid-folders]

# Clear title cache
rm -rf ~/quick_job_backend/vector_db/title_cache

# Clear all caches
rm -rf ~/quick_job_backend/vector_db
```

The server recreates all ChromaDB directories automatically on the next request.
